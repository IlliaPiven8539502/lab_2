## Опис системи

Реалізовано мінімальну еталонну архітектуру месенджера з розширенням **Варіанту 2 — відстеження статусів повідомлень**.

Система підтримує:
- створення користувачів
- створення розмов
- надсилання повідомлень
- перегляд історії розмови
- оновлення статусу повідомлення (`sent` → `delivered` → `read`)

### Архітектура

```
Client (браузер / Postman / curl)
        │
        ▼
   HTTP API (Router)          ← com.sun.net.httpserver (вбудований Java)
        │
        ▼
  Message Service             ← бізнес-логіка, валідація
        │
        ▼
    Database                  ← JSON Lines файловий storage
```

## API

| Метод | Маршрут | Опис |
|---|---|---|
| `POST` | `/users` | Створити користувача |
| `GET` | `/users` | Список усіх користувачів |
| `POST` | `/conversations` | Створити розмову |
| `GET` | `/conversations` | Список розмов |
| `POST` | `/messages` | Надіслати повідомлення |
| `GET` | `/conversations/{id}/messages` | Історія розмови |
| `PATCH` | `/messages/{id}/status` | Оновити статус (Варіант 2) |

## Персистентність даних

Дані зберігаються у форматі **JSON Lines** — кожен рядок файлу є окремим JSON-об'єктом:

```
# data/users.jsonl
{"id":"3f2a...","name":"Аліса"}
{"id":"9e1b...","name":"Богдан"}

# data/messages.jsonl
{"id":"7c8d...","conversationId":"a1b2...","senderId":"3f2a...","text":"Привіт!","createdAt":"...","status":"delivered"}
```

Переваги підходу:
- дані не губляться після перезапуску сервера
- не потребує зовнішніх залежностей (SQLite, PostgreSQL)
- файли читаються людиною без інструментів

## Обробка помилок

| Ситуація | HTTP-код | Відповідь |
|---|---|---|
| Порожній текст повідомлення | 400 | `{"error": "Текст повідомлення не може бути порожнім"}` |
| Користувач не існує | 400 | `{"error": "Користувача з id='...' не знайдено"}` |
| Розмова не існує | 400/404 | `{"error": "Розмову з id='...' не знайдено"}` |
| Невідомий статус | 400 | `{"error": "Невідомий статус '...'. Допустимі: sent, delivered, read"}` |
| Метод не підтримується | 405 | `{"error": "Метод не підтримується"}` |

## Технічні рішення

### Без зовнішніх залежностей

Проєкт не використовує жодних зовнішніх бібліотек — лише стандартна бібліотека Java:

| Потреба | Рішення |
|---|---|
| HTTP-сервер | `com.sun.net.httpserver.HttpServer` |
| JSON-серіалізація | ручна побудова рядків у методах `toString()` |
| JSON-парсинг | власний парсер у `Router.java` та `Database.java` |
| База даних | JSON Lines файли через `java.nio.file.Files` |
| UUID | `java.util.UUID.randomUUID()` |
| Timestamp | `java.time.Instant.now()` |

### Чому JSON Lines, а не SQLite?

SQLite потребує зовнішнього JDBC-драйвера. JSON Lines — простіший підхід який:
- не потребує залежностей
- зберігає дані між перезапусками (персистентність)
- файли читаються без спеціальних інструментів
- достатній для прототипу

## Відповіді на питання для захисту

**1. Як система гарантує збереження повідомлень?**  
Кожне повідомлення одразу записується у файл `messages.jsonl` через `Files.writeString` з режимом `APPEND`. Дані зберігаються на диску незалежно від стану сервера.

**2. Що відбувається якщо одержувач офлайн?**  
Повідомлення зберігається зі статусом `sent`. При наступному підключенні клієнт отримує його та надсилає підтвердження `delivered`. Статус оновлюється через `PATCH /messages/{id}/status`.

**3. Як повідомлення унікально ідентифікуються?**  
Кожне повідомлення отримує `id` згенерований через `UUID.randomUUID()` та `createdAt` у форматі ISO-8601 (`Instant.now().toString()`).

**4. Які помилки можуть виникнути при надсиланні?**  
Порожній текст, неіснуючий відправник, неіснуюча розмова — всі повертають HTTP 400 з описом помилки у полі `error`.

**5. Як система змінилася б для 1 мільйона користувачів?**  
- JSON Files → PostgreSQL з connection pool (HikariCP)
- Один сервер → кілька інстанцій за load balancer (nginx)
- Синхронна доставка → черга повідомлень (Kafka / RabbitMQ)
- Статуси через HTTP polling → WebSocket для real-time оновлень
- Додати кешування (Redis) для активних розмов
